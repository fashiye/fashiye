# 游戏代练交易平台 - 登录邮箱升级、打手分级、审批系统 - Product Requirement Document

## Overview
- **Summary**: 对现有游戏代练平台进行功能升级：将登录方式从用户名改为邮箱登录，允许用户名重复，打手新增游戏专属等级，后台将用户管理分为用户管理和打手管理，引入打手注册审批流程。
- **Purpose**: 提高用户体验（邮箱作为唯一登录凭证更符合习惯），实现打手按游戏的分级管理，加强平台对打手的准入控制和管理。
- **Target Users**: 玩家用户、打手、平台管理员

## Goals
- 将登录方式从用户名登录改为邮箱登录
- 允许用户名重复（邮箱保持唯一）
- 打手增加游戏专属等级系统，可在后台管理
- 将后台用户管理拆分为用户管理和打手管理两个独立模块
- 实现打手注册审批流程

## Non-Goals (Out of Scope)
- 不会改变用户支付和订单核心流程
- 不会修改游戏和项目管理功能
- 不会改变后台管理员系统
- 不会涉及用户密码加密方式变更

## Background & Context
- 当前系统：用户、打手、管理员使用统一登录，以用户名为唯一标识；打手有通用 level 字段；后台用户管理混在一起；打手注册无需审核。
- 技术栈：FastAPI + SQLAlchemy + React
- 数据库结构：users、handlers、admins 独立表，games、projects 等已有表
- 已有后端 API：/auth/login, /auth/register, /users 等

## Functional Requirements
- **FR-1**: 登录方式改为邮箱+密码登录，支持用户、打手、管理员使用邮箱登录
- **FR-2**: 允许用户名重复（保持邮箱唯一性），去除用户名字段 unique 约束
- **FR-3**: 新增 HandlersGameLevel 表，打手在每个游戏中有专属等级，后台可以管理打手等级
- **FR-4**: 后台用户管理拆分为用户管理（玩家）和打手管理两个独立模块
- **FR-5**: 打手注册需要管理员审批，初始状态为待审批，审批通过后才能登录
- **FR-6**: 后台打手管理支持审批、禁用、设置等级等操作
- **FR-7**: 更新游戏管理模块，关联打手等级信息

## Non-Functional Requirements
- **NFR-1**: 保持现有 API 响应性能，无明显性能下降
- **NFR-2**: 数据库变更需支持平滑迁移，不丢失现有数据
- **NFR-3**: 前端界面保持响应式，兼容现有设备

## Constraints
- **Technical**: 保持现有技术栈不变
- **Business**: 需保留现有用户和打手数据，兼容历史记录
- **Dependencies**: 依赖 SQLAlchemy ORM, FastAPI, React, 现有数据库表结构

## Assumptions
- 用户可以理解邮箱作为唯一登录标识
- 管理员会定期处理打手审批请求
- 现有游戏管理模块可以扩展关联打手等级

## Acceptance Criteria

### AC-1: 邮箱登录功能
- **Given**: 用户已注册且账号有效
- **When**: 用户使用邮箱+密码+角色进行登录
- **Then**: 系统验证成功后返回 token，用户可以正常登录
- **Verification**: `programmatic`

### AC-2: 用户名允许重复
- **Given**: 已有用户名为"张三"的用户
- **When**: 新用户使用同样的用户名"张三"但不同邮箱注册
- **Then**: 系统允许注册成功
- **Verification**: `programmatic`

### AC-3: 打手游戏专属等级
- **Given**: 有一个打手和多个游戏
- **When**: 后台管理员为打手设置各游戏的等级
- **Then**: 打手在每个游戏的等级被正确保存和显示
- **Verification**: `programmatic`

### AC-4: 用户管理和打手管理拆分
- **Given**: 管理员登录后台
- **When**: 管理员访问用户管理和打手管理
- **Then**: 看到两个独立的管理界面，各自对应玩家和打手数据
- **Verification**: `human-judgment`

### AC-5: 打手注册审批流程
- **Given**: 用户申请注册打手
- **When**: 注册成功但还未审批
- **Then**: 打手账号状态为待审批，无法登录，管理员在后台可以看到并审批
- **Verification**: `programmatic`

### AC-6: 后台打手管理功能
- **Given**: 管理员在打手管理页面
- **When**: 执行审批、禁用、设置等级等操作
- **Then**: 操作生效，状态变更正确
- **Verification**: `programmatic`
